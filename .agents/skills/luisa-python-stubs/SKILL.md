# LuisaComputeSimulator — Python Stubs Architecture

## The Problem

`lcs_py` is a C++ extension (`.so`) built with pybind11. It has **no
type information at the Python level**. `pybind11-stubgen` can extract
signatures from the compiled module, but some return types are `...`
and Pyright/Pylance may resolve `lcs_py` to the `.so` instead of the
`.pyi`, losing all completions and parameter hints.

## Architecture

```
pyrightconfig.json:
  stubPath = "PythonBindings/python"          ← Pyright highest priority
  extraPaths = ["PythonBindings/python", …]

PythonBindings/python/
├── lcs_py/                          ← Stub PACKAGE (stubPath priority)
│   ├── __init__.pyi                 ← Auto-generated by stubgen, copied here
│   └── py.typed                     ← PEP 561 marker for Pylance navigation
├── lcs/                             ← Runtime package
│   ├── __init__.py                  ← from lcs_py import *
│   ├── __init__.pyi                 ← Hand-maintained, class X(_X): …
│   └── py.typed                     ← PEP 561 marker
└── lcs_py.pyi                       ← Transient — stubgen output, renamed away
```

### Resolution Priority (Pyright)

1. **stubPath** → `lcs_py/__init__.pyi` (highest — bypasses the `.so` entirely)
2. extraPaths → `lcs_py.pyi` (fallback)
3. venv site-packages → `lcs_py.cpython-*.so` (no type info)

## Key Files

### `lcs/__init__.pyi` — Hand-Maintained Type Stub

Uses **subclassing** (not aliases) for type caching:

```python
from lcs_py import WorldData as _WorldData

class WorldData(_WorldData): ...   # ← cached local class, NOT alias
```

- `X = lcs_py.X` → type alias, Pyright re-traverses `lcs_py.pyi` every completion
- `class X(_X): ...` → local class definition, Pyright caches MRO — fast completions

**Enum values** (Cloth, Particle, Rigid, Rod, Tetrahedral) are NOT classes
in `lcs_py.pyi` — they're `MaterialType` aliases. Use plain `X = lcs_py.X`
for them, not subclassing.

**When to add overrides**: only when stubgen produces `...` for a type.
If the C++ side can expose the type (e.g., exposing `FixedPointsType` enum),
fix it there instead of adding an override.

### `lcs_py/__init__.pyi` — Stub Package

This is `pybind11-stubgen` output, copied into a package directory under
`stubPath`. The directory + `py.typed` tells Pyright this is a typed
package, which enables **"Go to Definition"** (Cmd+Click) navigation.

Without `py.typed` in `lcs_py/`, Pylance can complete but won't navigate
to the stub file.

## Regeneration Workflow

After changing C++ bindings (`PythonBindings/src/python_bindings.cpp`):

```bash
cmake --build build -j --target stubs
```

This target (defined in `PythonBindings/CMakeLists.txt`):
1. Sets `PYTHONPATH` to `build/bin/` so stubgen can import `lcs_py.so`
2. Runs `pybind11-stubgen lcs_py -o PythonBindings/python/`
3. Runs `normalize_stub.cmake` → moves output to `lcs_py/__init__.pyi`

### `normalize_stub.cmake`

Handles two output modes of pybind11-stubgen:
- `lcs_py.pyi` (single-file) → renames to `lcs_py/__init__.pyi`
- `lcs_py/__init__.pyi` (package mode) → already correct, no-op

Always ends with `lcs_py/__init__.pyi` existing.

## Critical Pitfalls

### 1. `py.typed` is required for navigation

Two `py.typed` files needed:
- `lcs/py.typed` — tells Pyright the `lcs` package is typed
- `lcs_py/py.typed` — tells Pyright/Pylance to navigate into the stub
  package for "Go to Definition"

### 2. Stub staleness

`lcs_py.pyi` is NOT auto-generated by cmake. `pybind11-stubgen` is a
separate tool. After C++ binding changes, you MUST run the `stubs`
target. The `.pyi` does not auto-update.

### 3. `lcs_py.pyi` is transient

The `stubs` target renames `lcs_py.pyi` → `lcs_py/__init__.pyi`. After
running `--target stubs`, `lcs_py.pyi` no longer exists — only
`lcs_py/__init__.pyi` remains. This is by design (no duplication).
